Reliable low-latency UDP playout

Flaky Network systems assignment - two-frame XOR forward-error correction

This system-level summary was created by a coding agent from the implementation and the recorded experiment results.

Architecture

Harness source :47010 → sender :47001 → hostile relay :47002 → receiver :47020 → harness player

The harness produces one 160-byte payload every 20 ms. The relay can drop, delay, duplicate, and reorder UDP packets. The player accepts a frame only when the correct sequence number and payload arrive before that frame's playout deadline.

Sender

The sender binds UDP port 47010 and reads the harness format: a four-byte big-endian sequence number followed by a 160-byte payload. It immediately transmits a data packet to the relay so normal frames take only one relay crossing. For each pair of consecutive frames, it calculates the bytewise XOR of their payloads and transmits one additional parity packet after the second frame arrives. It uses only standard C socket interfaces and talks only to the relay media port.

Data packets contain a packet-type byte, the original sequence number, and the payload. Parity packets contain a packet-type byte, the pair's first sequence number, and the 160-byte XOR value. The sender does not use a side channel or direct process-to-process communication.

Receiver

The receiver binds UDP port 47002 and maintains bounded slots indexed by two-frame block base sequence number. A directly received frame is stored and forwarded to the harness player on port 47020 immediately. Duplicate data packets are ignored after the first copy, so relay duplication does not cause repeated work or alter recovery state.

When the receiver has a parity packet and exactly one missing data frame in the pair, it reconstructs the missing payload by XORing the parity value with the received payload. It then emits the recovered frame in the exact harness format. Because the reconstruction is based on explicit sequence numbers, packets may arrive in either order without corrupting the frame mapping.

Trade-offs and timing

One parity packet for every two data packets gives a measured upstream overhead of 1.55x, below the 2.0x limit. It can recover one loss in a pair, including a dropped data packet or a delayed packet that misses its deadline. It cannot recover two losses in the same pair, or simultaneous loss of a data packet and its parity packet.

A parity packet is available only after the second frame of a pair is produced, 20 ms after the first. With profile B's delay range of up to 80 ms, a lost first frame can require roughly 100 ms before its recovery packet reaches the receiver. This is why lower delay settings become unreliable even though the code forwards direct data immediately.
